## **Problem definition and context**

A SCADA/MES system provides data from machines, this data could include production counts, machine component values, error codes and much more. With a SCADA system this data can be interpreted and supervised and understood to give much more insight and understanding to the machine. It can be used to track performance and analyse trends in waste or machine components deviation from average values. SCADA/MES systems allow engineers to understand machines on a deeper level and allow issue to be identified the day of occurrence rather than having to monitor machines for a period instead. These systems allow a whole new different level of understanding of machines and contribute to troubleshooting and ensuring machines are available to run for as much time as possible.
	The current system used by Hollister ULC, called MI, is not suitable for this plant. It provides data on the machines and allows users to interpret the data, but incurs a 15-minute delay, and its data is not real time. MI does not provide a user-friendly interface, and it is not a straightforward and effortless process, the interface is not modern or advanced and looks outdated and complicated for the user to navigate. The capabilities of this system do not match the characteristics required, a system like this would be expected to be used every day by engineers to understand what the machine has done in the previous 24 hours and what it is doing right now. When trying to use MI for this, there is a lag that could take up to 30 minutes to load, ~ 6.5% of an engineer’s day is spent waiting for a system to reply. Once MI has loaded there will then be another delay in formatting graphs and receiving and interpreting the information, this totals approximately one whole hour of an engineer’s day (~12.5%). Obviously, this is exceedingly far from suitable, when you incorporate this with the old and uncomfortable user-interface, this system clearly is not suitable for its needs. MI is useful for generating long term data and graphs for reports over prolonged periods, but for daily information that needs to be fast, easily interpretable and provides short term data, MI is not suitable. In terms of industry, to keep up with the industry standard, there is no avoiding the fact that MI needs to be upgraded or changed.


## **Objectives and methodology**

To counter this problem, I decided to create a real time, machine specific SCADA system that will provide data on a machine and allow engineers to retrieve data on the machine instantly with little to no delay. A system that can provide data to the engineers and help them understand exactly why machine defects have started to rise is exceptionally useful and will provide information to engineers that previously wasn’t available. A successful system would provide real time data on the machine – running/faulted, production numbers, error codes, machine component information, an audit trail tracking parameter changes, and a general OEE result and breakdown into Availability, Performance and Quality. The system will provide today and yesterdays data and will provide a small historian but not a massive one, as that is not the targeted use of this project. MI still works for old data and when users have time to generate data and graphs it is a feasible option and provides a full picture, therefore that is not the targeted use of the new system. A short historian will be included to identify trends and waste spikes but not for long term trend analysis on rejects etc.
	The software I will use to produce this system will be Ignition by Inductive Automation. It offers a quick service and allows poling of PLC tags at a custom frequency. It operates off a client (Website) – gateway (Server) – device (Machine PLC) set up, where you can have multiple clients and devices however all data is sorted through the gateway. My SCADA site will be on the client end of the setup and will act similar to an interactive dashboard displaying different stats and metrics about the machine. I will create a website like page that will be interactive allow the user to manipulate the data being received from the PLC and display short term graphs and tables. The overall goal of this site is to provide quick easy and accessible data for engineers, which includes relevant and specific data, ands allows them to troubleshoot the machine and reduce waste and downtime and ultimately increase machine’s good parts output. 
	The site will consist of different pages which will show different metrics on the machine. To break down the information down into major sectors, there would be machine downtime (Faults, error codes), Machine Quality (number of rejects) and Performance. These 3 metrics will be the whole basis on judging whether the machine performed successfully in the previous 24 hours. By using these three metrics the OEE can be calculated. This single number can be used to compare different machines and understand the general level of the machine performance against other benchmarks. With this there is so much data that can be extracted for each metric. For quality, each reject can be tracked and displayed with each reason, from there the values can be trended on a graph. I have a graph displaying the number of rejects by station, this provides extremely useful information for the engineers as wherever the rejects are occurring this can then be targeted by the engineers to reduce the quantity of waste being created throughout out the machine and improve overall performance.
